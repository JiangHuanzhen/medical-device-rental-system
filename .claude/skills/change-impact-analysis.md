# 变更影响分析 (CIA)

## 操作定义
在变更请求被接受后、代码修改开始前，从九个维度系统化评估变更的波及范围。CIA 的价值在于将变更从"改了再说"拉回"先想清楚再动手"——明确变更影响了什么、不影响什么，防止在代码中到处打补丁。

## 参数说明
| 参数 | 默认值 | 说明 |
|------|--------|------|
| `CR_PATH` | — | 变更需求文档路径（必填） |
| `BASELINE_DIR` | `wiki/baselines/BL-*-latest/` | 当前基线目录 |
| `DESIGN_DIR` | `wiki/design/` | 设计产物目录 |
| `SRC_DIR` | `project/src/` | 当前源代码目录 |

## 执行步骤

### 9维度逐项评估
对每个维度标注受影响程度（无/轻微/中等/严重）并给出具体说明：

1. **需求层面**: SRS 中哪些需求条目受影响？是否需要更新 RTM 溯源矩阵？
2. **架构层面**: 是否影响 ASD 架构约束或分层规则？是否需要新增/修改 ADR？
3. **模块层面**: 涉及哪些模块的修改？`depends_on` 是否变化？是否需要新增模块？
4. **契约层面**: OpenAPI 接口的请求体/响应体/错误码是否变化？是否需要递增 `x-version`？
5. **代码层面**: 需要修改哪些源文件？新增哪些文件？删除哪些文件？
6. **数据层面**: 数据库 Schema/索引/缓存策略是否需要变更？是否有数据迁移需求？
7. **兼容性层面**: 旧版本客户端是否仍能正常工作？是否需要 API 版本升级？是否存在 `deprecated` 字段？
8. **测试层面**: 哪些已有测试用例会受影响？需新增哪些测试用例？
9. **部署运维层面**: CI/CD 配置/监控告警/部署顺序是否需要调整？

### 影响路径追踪
- 从变更点出发，沿 DTS 依赖拓扑正向追溯受影响的模块
- 沿 RTM 反向追溯到受影响的需求条目
- 输出影响路径图（文本形式）

### 风险识别
- 识别变更可能引入的回归风险
- 标注高风险区域（耦合度高、测试覆盖低、近期频繁变更的模块）

## 输出格式
```markdown
# 变更影响分析 (CIA) 报告

## 变更概述
- 变更来源: {来源}
- 当前行为: {当前}
- 期望行为: {期望}
- 变更优先级: {优先级}

## 9维度评估表
| 维度 | 受影响程度 | 具体说明 | 建议措施 |
|------|----------|---------|---------|

## 影响路径
{文本形式的影响传播路径}

## 风险矩阵
| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|

## 变更实施建议
1. 推荐实施顺序
2. 回归校验重点
3. 回滚方案

## 需要更新的产物
| 产物 | 更新内容 | 优先级 |
|------|---------|--------|
```

## 质量标准
- 9个维度全部评估，无遗漏
- 每个「中等」及以上的影响有对应的缓解措施
- 影响路径可追溯到 DTS 依赖边
- 实施建议具体到文件级别

## 验证步骤
1. 确认影响维度没有只写「无」而不给出分析过程
2. 抽查一条影响路径是否与 DTS 拓扑一致
3. 确认风险缓解措施不是空话（有具体行动）
