# 质量校验提示词模板

## 适用场景
对四种设计产物的质量进行四维度校验，适用范围：任何产出 ASD/MDS/DTS/ADR 的分层架构项目。

## 使用方法
选择校验维度，替换 `{DIMENSION}` 和路径参数后调用 LLM。

## 完备性校验模板
```
你是工程产物完备性审计师。检查以下设计产物是否存在遗漏。

## ASD
{ASD}

## MDS
{MDS}

## DTS
{DTS}

## ADR
{ADR}

## 检查项
1. 逐条检查功能需求是否在 MDS 中有对应模块承担
2. 检查每个模块的每项职责在 DTS 中是否有合法依赖边支撑
3. 检查 ADR 是否覆盖了所有关键架构决策点（选型/技术栈/数据库/部署）

## 输出格式
| 问题编号 | 产物 | 遗漏内容 | 严重程度 | 建议 |
```

## 正确性校验模板
```
你是工程产物正确性审计师。检查以下设计产物是否存在语义错误。

## 检查项
1. 检查每条约束条款的措辞是否有歧义（含「尽量」「尽可能」「通常」「适当」等模糊词）
2. 检查每个模块的职责描述是否真正做到了单一职责（不含「和」字）
3. 检查 DTS 中每条边的类型是否正确地反映了功能交互的性质

## 输出格式
| 问题编号 | 产物 | 错误描述 | 严重程度 | 修正建议 |
```

## 一致性校验模板
```
你是工程产物一致性审计师。执行四组交叉比对。

## 四组交叉比对
1. ASD vs MDS: ASD 的分层规则是否与 MDS 的模块组织一致？
2. ASD vs DTS: ASD 的约束条款是否在 DTS 的 forbidden_dependencies 中体现？
3. MDS vs DTS: MDS 的模块是否都在 DTS 中有对应节点和合法依赖？
4. ADR vs 其他: ADR 的决策内容是否与其他产物的实际内容一致？

## 输出格式
| 矛盾编号 | 交叉组 | 产物A内容 | 产物B内容 | 矛盾描述 | 严重程度 |
```

## 有效性校验模板
```
你是工程产物有效性审计师。评估这些产物能否有效驱动AI代码生成。

## 检查项
1. 逐条检查约束条款是否「可判定」——包含明确的判断条件，不依赖主观判断
2. 选择三个典型代码生成场景测试约束信息是否充分

## 三个测试场景
- 场景A（跨模块调用）: 生成业务层代码时，ASD+TLCD是否约束了合法调用链？
- 场景B（数据访问）: 生成数据层代码时，TLCD是否禁止包含业务逻辑？
- 场景C（异常处理）: 生成全局异常处理时，C-CODE是否约束了异常处理方式？

## 输出格式
| 问题编号 | 约束编号 | 不可判定原因 | 缺失信息 | 建议 |
```

## 不适用场景
- 非正式工程产物（草稿阶段的文档不应进行严格校验）
- 单文件项目（无需模块间一致性检查）
