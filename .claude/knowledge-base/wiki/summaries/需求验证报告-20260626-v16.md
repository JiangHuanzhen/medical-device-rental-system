# 需求验证报告

验证日期：20260626
总体结论：**获取类问题**
发现 7 个问题

## 发现清单（教材§7四类交叉比对）
| 类型 | 严重程度 | 涉及章节 | 问题描述 | 修正建议 |
|-----|---------|---------|---------|---------|
| 对话偏差 | 重要 | 3.1.2 FR-EQP-004 | SRS中FR-EQP-004的默认校准承诺时限为24小时，与涉众对话记录存在偏差。运维工程师（BR-EQP-009）明确建议默认SLA保持72小时，而招商业务员（BR-EQP-008）建议24小时。SRS直接采用了招商业务员的建议，未体现运维工程师的需求，也未说明为何选择24小时而非72小时。 | 建议回A2分析，明确默认校准承诺时限的最终决策依据。若采用24小时，需在SRS中说明理由，并评估对运维工程师72小时建议的覆盖情况。 |
| 对话偏差 | 重要 | 3.1.2 FR-EQP-005 | SRS中FR-EQP-005的默认审批流程顺序为“科室主任 → 库房主管 → 质控部门”，与涉众对话记录存在偏差。财务（BR-EQP-013）明确要求“库房主管的签字是必须参与的”，但未提及科室主任和质控部门的顺序。运维工程师（BR-EQP-014）提出“审批流程也应有所区分”，但SRS未体现根据发起人角色（库房人员/运维工程师）区分审批流程。 | 建议回A2分析，明确审批流程的最终设计。需确认默认顺序的合理性，并评估是否需要根据发起人角色设置不同的审批流程。 |
| 对话偏差 | 建议 | 3.1.2 FR-EQP-004 | SRS中FR-EQP-004的业务规则提到“对于大型精密设备，默认校准承诺时限为72小时，但需特殊审批”，这与财务（BR-EQP-010）的建议“一般设备强制24小时，大型精密设备可放宽至72小时，但需特殊审批”在语义上等价，但SRS未明确“特殊审批”的具体流程或条件，存在模糊性。 | 建议在SRS中明确“特殊审批”的具体含义，例如是否需要更高级别的审批人，或需要额外的审批材料。 |
| 对话偏差 | 建议 | 3.1.2 FR-EQP-005 | SRS中FR-EQP-005的业务规则提到“系统支持‘应急越签’模式，在紧急情况下，可跳过部分节点，但必须由发起人申请并说明理由”，该功能在涉众对话记录中未被提及。这是一个新增的功能点，属于“合理诠释”范畴，但未在SRS中说明其来源，可能引入未经验证的风险。 | 建议回A2确认“应急越签”模式是否为必要功能，并补充其触发条件、审批规则和风险控制措施。 |
| 文档矛盾 | 建议 | 2.3 用户角色与特征 | SRS中“库房主管”角色的职责包含“执行锁库操作”，但根据FR-EQP-002，锁库操作是由系统定时任务自动触发的，而非人工执行。SRS的角色定义与功能需求描述存在矛盾。 | 建议修改“库房主管”的职责描述，将“执行锁库操作”改为“接收锁库通知并确认”，或明确其是否拥有手动触发锁库的权限。 |
| 文档矛盾 | 建议 | 2.3 用户角色与特征 | SRS中“质控部门”的职责包含“审批紧急出库”，但在FR-EQP-005的审批流程中，质控部门是审批节点之一。然而，在涉众对话记录中，质控部门并未直接参与紧急出库审批的讨论。SRS将质控部门纳入审批流程，但未在涉众对话记录中找到直接证据支持。 | 建议回A2确认质控部门在紧急出库审批中的角色和职责，确保其参与的必要性和合理性。 |
| 内部不一致 | 建议 | 3.4 数据字典 | SRS数据字典中`Device`表的`status`字段的默认值为`'在库'`，但在`Emergency_Outbound`表的`status`字段中，状态值包含`'待审批'`、`'审批通过'`等。SRS未定义`Device`表`status`字段中`'紧急出库'`状态的具体含义，也未说明设备在紧急出库审批通过后，其状态如何从`'已锁库'`或`'禁止出库'`转变为`'紧急出库'`。 | 建议在数据字典中补充`Device`表`status`字段的完整枚举值列表，包括`'紧急出库'`，并明确状态转换的规则。 |
