# 需求验证报告

验证日期：20260626
总体结论：**获取类问题**
发现 7 个问题

## 发现清单（教材§7四类交叉比对）
| 类型 | 严重程度 | 涉及章节 | 问题描述 | 修正建议 |
|-----|---------|---------|---------|---------|
| 对话偏差 | 重要 | 3.1.2 FR-EQP-002 | SRS中紧急出库审批流程的第四级为“财务人员”，与涉众原话不符。涉众（库房人员）明确提出的顺序是“库房主管 → 质控部门 → 科室主任”，并未提及财务人员。SRS增加了财务审批环节，属于对涉众需求的扩展，但未在需求清单中找到对应依据。 | 请A2分析工程师确认：增加“财务人员”作为第四级审批是否为新的、已确认的需求？如果是，需补充相关涉众对话记录或需求条目。如果不是，应修正SRS，移除财务审批环节，或将其作为可选/并行节点。 |
| 对话偏差 | 重要 | 3.1.2 FR-EQP-003 | SRS中归还检测流程的“数据比对与处理”步骤描述为“系统阻断‘完成收回’按钮，弹出提示...”，这与运维工程师的明确建议相悖。运维工程师（BR-EQP-008）明确建议“系统不应直接‘拦住不让入库’，而应该实施‘分级预警+流程触发’机制”。SRS的描述是“直接硬性阻止”的一种形式，属于对涉众需求的曲解。 | 请A2分析工程师重新审视此流程。建议将“阻断操作”改为“分级预警”，例如：偏差超出范围时，系统弹出强警告并提示“检测数据异常，建议创建维修工单”，但仍允许用户选择“强制入库（需填写原因）”或“创建维修工单”，而非直接阻断。 |
| 对话偏差 | 重要 | 3.1.2 FR-EQP-003 | SRS中归还检测流程的“复核”步骤描述为“库房人员对所有检测项进行全面、独立的复核”，这与涉众（库房人员）的原话存在偏差。涉众（BR-EQP-006）强调“双人双责”，但并未明确要求库房人员必须“对所有检测项”进行复核。SRS的描述可能过度设计，增加了不必要的操作负担。 | 请A2分析工程师与涉众确认：库房人员的复核范围是“所有检测项”还是“系统标记的可疑项”？根据运维工程师（BR-EQP-008）的“分级预警”思路，复核范围可能更侧重于异常项。请澄清后修正SRS。 |
| 对话偏差 | 重要 | 3.1.2 FR-EQP-001 | SRS中校准预警的推送频率描述为“每周一上午10:00”，与涉众（运维工程师）的原话“每周一早上推送一次”存在细节偏差。涉众并未指定具体时间点，SRS将其具体化为“上午10:00”，属于合理诠释。但需确认该时间点是否与业务习惯冲突。 | 建议将此时间点作为可配置项，由系统管理员在“系统配置”模块中设定，以增加灵活性。 |
| 文档矛盾 | 重要 | 3.1.2 FR-EQP-002 | SRS中紧急出库审批流程的参与角色包含“库房主管”，但用户角色定义（2.3节）中并无“库房主管”角色，仅有“库房人员”。角色定义与功能需求描述不一致。 | 请A2分析工程师确认：是否需要新增“库房主管”角色？如果是，需在2.3节补充其职责和权限。如果不是，应将FR-EQP-002中的“库房主管”统一修改为“库房人员”。 |
| 内部不一致 | 建议 | 3.1.2 FR-EQP-001 | SRS中FR-EQP-001的业务规则提到“已出租”状态指“设备已出库并完成客户验收确认”，但数据字典（3.2节）中EQUIPMENT表的status字段说明为“设备状态（在库/已出租/维修中/校准过期等）”，未对“已出租”状态进行明确定义。功能需求中的定义未在数据字典中体现。 | 建议在数据字典的EQUIPMENT.status字段说明中，补充“已出租”状态的定义，使其与功能需求保持一致。 |
| 内部不一致 | 建议 | 3.1.2 FR-EQP-003 | SRS中FR-EQP-003的业务规则提到“归还检测必须使用与入库检测完全一致的标准化检查清单和模板”，但功能需求列表（3.1.2节）中并未包含“入库检测”相关的功能需求（FR）。这导致“与入库检测完全一致”的规则缺乏对应的功能实现作为参照。 | 建议在SRS中补充“入库检测”相关的功能需求（FR），或至少在FR-EQP-003中明确说明“入库检测模板”由哪个功能需求（如FR-CFG-001）进行管理，以建立追溯关系。 |
